## 1. 安全性

### 通用检查项

**输入验证**
- [ ] 所有来自外部的输入（HTTP参数、文件内容、数据库读取、消息队列、RPC入参）是否在使用前验证？
- [ ] 是否存在 SQL 注入风险（字符串拼接 SQL，未使用参数化查询）？
- [ ] 是否存在命令注入（将用户输入传给 Runtime.exec / subprocess / eval）？
- [ ] 文件路径是否做了规范化和白名单校验（防止路径穿越 `../../etc/passwd`）？
- [ ] XML 解析是否禁用了外部实体（防 XXE）？
- [ ] 反序列化是否对类型做了白名单限制？

**敏感数据**
- [ ] 密码、API Key、Token、私钥是否硬编码在代码中？
- [ ] 敏感字段（密码、身份证、银行卡）是否出现在日志中？
- [ ] HTTP 响应中是否意外暴露了内部系统信息（堆栈、内部 IP、版本号）？
- [ ] 数据库中的敏感字段是否加密存储？传输是否加密？

**认证与授权**
- [ ] 新增的 API 接口是否有身份认证保护？
- [ ] 是否有水平越权风险（用户 A 能访问用户 B 的数据，仅因为知道 ID）？
- [ ] 是否有垂直越权风险（普通用户能调用管理员接口）？
- [ ] Session / Token 的生成是否使用了安全随机数？
- [ ] 密码重置、邮箱验证等敏感操作是否有防重放保护？

**加密**
- [ ] 是否使用了已知弱算法（MD5/SHA1 用于密码存储；DES/RC4 用于加密）？
- [ ] 对称加密是否正确使用了 IV（初始化向量）且 IV 随机？
- [ ] 随机数生成是否使用了密码学安全的 CSPRNG？

**Web 特定（如果涉及 Web 开发）**
- [ ] 输出到 HTML 是否做了 HTML 实体编码（防 XSS）？
- [ ] 状态变更操作是否有 CSRF 防护？
- [ ] 文件上传是否校验了文件类型（不仅是扩展名，还有 MIME/Magic bytes）？
- [ ] 重定向地址是否做了白名单校验（防开放重定向）？

**严重程度参考**：
- P0：直接可利用的注入漏洞、认证绕过、敏感数据明文泄露
- P1：潜在越权、弱加密算法、未完整校验的输入
- P2：日志中包含非必要敏感信息、缺少安全响应头

## 2. 资源管理

### 通用检查项

**文件与 IO**
- [ ] File、InputStream、OutputStream、Reader、Writer 是否在 finally 或 try-with-resources 中关闭？
- [ ] 文件读取是否有大小限制（防止超大文件导致 OOM）？
- [ ] 临时文件是否在使用后删除？
- [ ] 文件操作是否考虑了磁盘空间不足的情况？

**数据库连接**
- [ ] Connection、Statement、ResultSet 是否正确关闭？
- [ ] 是否使用了连接池？连接池配置是否合理（最大连接数、超时时间）？
- [ ] 事务是否正确提交/回滚？是否存在忘记关闭事务的路径？
- [ ] 是否在循环中频繁获取/释放连接（应批量处理）？

**网络连接**
- [ ] HTTP Client、Socket 是否设置了超时（连接超时 + 读取超时）？
- [ ] 是否使用了连接池避免重复建立 TCP 连接？
- [ ] 是否处理了连接中断和超时的重试逻辑？

**内存**
- [ ] 是否将大量数据全量加载到内存（应改为流式/分页处理）？
- [ ] 长生命周期的集合（如 static Map）是否有清理机制（防内存泄漏）？
- [ ] 是否有意外持有大对象引用导致 GC 无法回收？
- [ ] 递归调用是否有深度限制（防栈溢出）？

**线程/进程**
- [ ] 新建的线程是否有适当的命名（便于排查）？
- [ ] 是否使用了线程池而非直接 new Thread？
- [ ] 线程池的拒绝策略是否合理？
- [ ] 定时任务是否会因前一次未完成导致任务堆积？

**严重程度参考**：
- P0：无超时的远程调用（可能导致线程耗尽）、循环中创建大对象
- P1：未关闭的文件/连接（长期运行必然泄漏）、无上限的内存增长
- P2：未使用连接池、合理但非最优的资源使用方式

## 3. 异常处理

### 通用检查项

**捕获粒度**
- [ ] 是否捕获了过于宽泛的异常（Exception、Throwable、BaseException）而丢失了具体错误信息？
- [ ] catch 块是否为空（吞异常）？或只打了日志而没有任何处理？
- [ ] checked exception 是否被无意义地包装成 RuntimeException 而丢失堆栈？

**异常信息**
- [ ] 日志是否记录了异常发生时的关键上下文（用户ID、操作参数、请求ID）？
- [ ] 是否同时记录了异常对象本身（`log.error("msg", e)` 而非 `log.error("msg" + e.getMessage())`）？
- [ ] 对外返回的错误信息是否暴露了内部细节（堆栈、SQL、内部路径）？

**异常传播**
- [ ] 底层异常是否正确封装并传播给调用方，而不是仅记录后返回 null 或默认值？
- [ ] 异常处理后的返回值是否会被上层误判为成功（如返回 null 代替抛异常）？
- [ ] finally 块中是否有可能覆盖原始异常的代码？

**资源清理**
- [ ] 异常路径上是否有资源未释放（连接、锁、临时文件）？
- [ ] finally 块本身是否可能抛出异常（会覆盖原始异常）？

**重试与降级**
- [ ] 对可重试的异常（网络超时、限流）是否有重试机制？
- [ ] 重试是否有次数限制和退避策略（避免雪崩）？
- [ ] 关键操作失败时是否有降级方案？

**严重程度参考**：
- P0：吞掉关键操作（如支付、写入）的异常，导致数据不一致
- P1：资源泄漏的异常路径、对外暴露内部错误信息
- P2：日志信息不足导致难以排查、过宽的 catch 粒度
- P3：异常命名不规范、缺少必要注释

## 4. 并发安全

### 通用检查项

**共享状态**
- [ ] 多线程共享的变量是否有同步保护（synchronized / volatile / atomic / lock）？
- [ ] 是否使用了线程不安全的集合（HashMap、ArrayList）在多线程场景？
- [ ] 单例模式的懒加载是否线程安全（double-checked locking 是否正确）？
- [ ] 复合操作（check-then-act、read-modify-write）是否原子化？

**锁的使用**
- [ ] 是否存在死锁风险（多个锁的获取顺序是否一致）？
- [ ] 锁的粒度是否合理（锁的范围不要过大；也不要过细导致频繁竞争）？
- [ ] 锁是否在异常路径上也能正确释放（使用 try-finally）？
- [ ] 是否使用了更高级的并发原语（ReentrantLock、ReadWriteLock）替代粗粒度 synchronized？

**数据库并发**
- [ ] 是否存在并发写同一行的竞争条件（应使用乐观锁或数据库锁）？
- [ ] 乐观锁的重试逻辑是否正确实现？
- [ ] 分布式场景下是否需要分布式锁（Redis、ZooKeeper）？

**缓存并发**
- [ ] 缓存更新时是否存在 race condition（先删缓存还是先更新DB）？
- [ ] 缓存失效时是否存在缓存击穿（大量请求同时打到 DB）？
- [ ] 是否有缓存雪崩风险（大量 key 同时过期）？

**异步/消息**
- [ ] 异步任务的结果是否正确等待（避免 fire-and-forget 导致错误无感知）？
- [ ] 消息消费是否幂等（消息重复消费是否安全）？
- [ ] 异步操作的上下文（用户信息、Trace ID）是否正确传递？

**严重程度参考**：
- P0：明确的竞争条件导致数据损坏、死锁风险
- P1：线程不安全的集合在多线程场景、未原子化的复合操作
- P2：锁粒度不当导致性能问题、缺少幂等保护
- P3：可以优化的锁策略、建议使用更合适的并发原语

## 5. 兼容性

### 通用检查项

**API 兼容性**
- [ ] 已对外发布的接口是否删除了参数或修改了参数类型？
- [ ] 是否删除了枚举值（下游可能已经序列化了旧值）？
- [ ] 是否修改了返回值结构（删除字段、修改字段类型）？
- [ ] 是否修改了接口的行为语义（相同输入不同输出）？
- [ ] 新增的必填参数是否破坏了旧版调用方？

**数据库兼容性**
- [ ] 是否删除了正在使用的数据库列？
- [ ] 是否修改了列的数据类型？
- [ ] NOT NULL 的新增列是否有默认值（否则无法向已有数据插入）？
- [ ] 是否有大表的 DDL 操作（可能锁表）？
- [ ] 索引变更是否会影响现有查询的执行计划？

**配置兼容性**
- [ ] 是否删除或重命名了配置项（需要迁移指南）？
- [ ] 新增的必填配置项是否有合理的默认值？
- [ ] 是否修改了配置的取值范围或含义？

**序列化兼容性**
- [ ] 修改了可序列化的类（Java Serializable / Protobuf / Avro）是否保持向后兼容？
- [ ] Protobuf 是否删除了字段或修改了字段编号？
- [ ] JSON 序列化的类是否有新增非 optional 字段影响反序列化？

**依赖兼容性**
- [ ] 升级的依赖是否有 Breaking Changes（查阅 Changelog）？
- [ ] 降级的依赖是否丢失了正在使用的特性？
- [ ] 新增依赖是否与现有依赖存在版本冲突？
- [ ] 新增依赖是否有已知安全漏洞（CVE）？

**严重程度参考**：
- P0：删除正在使用的 API 字段/枚举/接口、无默认值的 NOT NULL 新列
- P1：Breaking API 变更、不兼容的序列化修改
- P2：配置项变更无迁移说明、依赖有已知 CVE
- P3：建议保留废弃标记（@Deprecated）而非直接删除

## 6. 性能

### 通用检查项

**算法与数据结构**
- [ ] 是否引入了 O(n²) 或更高复杂度的算法（嵌套循环遍历大集合）？
- [ ] 数据结构选择是否合适（频繁查找用 Map/Set 而非 List 遍历）？
- [ ] 排序是否可以避免（数据库侧排序 vs 内存排序）？

**数据库**
- [ ] 是否在循环中执行数据库查询（N+1 问题）？
- [ ] 批量操作是否使用了 batch insert/update？
- [ ] 查询结果集是否过大（应使用 LIMIT 或分页）？
- [ ] 新增的查询条件是否有对应索引？
- [ ] 是否使用了 `SELECT *`（应指定具体字段）？

**缓存**
- [ ] 热点数据是否使用了缓存？
- [ ] 缓存是否有合理的过期策略？
- [ ] 是否存在缓存使用后未命中就频繁穿透 DB 的场景？

**网络与 RPC**
- [ ] 是否在循环中进行远程调用（应批量或并行）？
- [ ] 响应数据是否存在不必要的大字段序列化？
- [ ] 是否可以使用异步调用减少等待时间？

**字符串与对象**
- [ ] 循环中是否有字符串 + 拼接（应使用 StringBuilder）？
- [ ] 是否频繁创建大对象（应考虑对象池或复用）？
- [ ] 正则表达式是否在循环内编译（应预编译为 Pattern）？

**严重程度参考**：
- P0：（性能问题通常不是 P0，除非导致生产超时/OOM/故障）
- P1：N+1 查询在高频路径上、循环中的远程调用
- P2：缺少索引、全量加载大数据集、未使用批量 API
- P3：小范围的优化建议（如 StringBuilder 替换）

## 7. 代码规范

### 通用检查项

**命名**
- [ ] 变量/方法/类名是否清晰表达其用途（避免 a、tmp、data 等无意义名称）？
- [ ] 命名是否与业务术语一致（使用统一的领域语言）？
- [ ] 布尔变量/方法是否以 is/has/can 等前缀命名？
- [ ] 常量是否使用全大写加下划线（UPPER_SNAKE_CASE）？

**函数设计**
- [ ] 函数是否超过 50 行（需要考虑拆分）？
- [ ] 函数是否符合单一职责原则（只做一件事）？
- [ ] 参数是否超过 5 个（考虑封装成对象）？
- [ ] 是否有多层嵌套（超过 3 层应考虑提前返回或抽取方法）？

**代码质量**
- [ ] 是否有明显的代码重复（DRY 原则违反）？
- [ ] 是否有魔法数字/字符串（应定义为具名常量）？
- [ ] 是否有大量注释掉的代码（应删除，版本控制会保留历史）？
- [ ] 未完成的 TODO 是否已经超期（需要处理或记录 issue）？

**注释**
- [ ] 复杂的业务逻辑是否有注释解释"为什么"而非"做了什么"？
- [ ] 注释是否与代码保持同步（代码改了但注释未更新）？
- [ ] Public API 是否有文档注释（Javadoc / JSDoc / docstring）？

**测试**
- [ ] 新增的业务逻辑是否有对应的单元测试？
- [ ] 测试是否覆盖了边界条件（空值、边界值、异常路径）？
- [ ] 测试是否有意义（不只是 happy path，不只是 mock 所有依赖）？
- [ ] 测试是否独立（不依赖执行顺序、不依赖外部环境）？

**严重程度参考**：
- P1：生产代码中的 `System.out.println` / `print` 调试语句
- P2：函数过长（100行+）、严重命名混乱影响可维护性
- P3：命名建议、注释补充、测试覆盖建议

## 8. 语言特定规则

### Java

**安全**
- 使用 `PreparedStatement` 而非 `Statement` 拼接 SQL
- 反序列化使用 `ObjectInputFilter` 白名单
- `Random` → `SecureRandom`（安全场景）

**资源管理**
- 优先使用 try-with-resources（Java 7+）
- 使用 `@Cleanup`（Lombok）或 `Closeable.close()`
- `Stream` 操作后记得关闭（如 `Files.lines()`）

**并发**
- 优先使用 `java.util.concurrent` 包的高级原语
- `volatile` 仅保证可见性，不保证原子性
- `HashMap` → `ConcurrentHashMap`（多线程）
- 避免 `synchronized(this)` 粒度过粗

**性能**
- 字符串拼接用 `StringBuilder`（循环中）
- 使用 `String.isEmpty()` 而非 `"".equals(s)`
- `ArrayList` vs `LinkedList`：随机访问用前者，频繁插删用后者
- 使用 Stream 并行流要注意线程安全

**代码规范**
- 使用 `Optional` 而非返回 null（避免 NPE）
- 使用枚举替代整型常量
- 合理使用泛型，避免原始类型

---

### Python

**安全**
- 避免 `eval()`、`exec()` 执行用户输入
- 使用参数化查询（`cursor.execute("SELECT ? FROM t", (val,))`）
- 文件路径使用 `pathlib.Path` 并验证是否在允许目录内

**资源管理**
- 使用 `with` 语句管理文件、连接等资源
- 使用 `contextlib.contextmanager` 自定义上下文管理器
- 生成器处理大数据集，避免全量加载

**并发**
- GIL 限制：CPU 密集型用 `multiprocessing`，IO 密集型用 `threading` 或 `asyncio`
- 异步代码中避免阻塞调用（使用 `asyncio.run_in_executor`）
- 使用 `threading.local()` 存储线程本地数据

**性能**
- 列表推导式通常比等价的 `for` 循环更快
- 使用 `set`/`dict` 而非 `list` 做成员查找（O(1) vs O(n)）
- 避免全局变量（访问局部变量更快）

---

### Go

**安全**
- 使用 `database/sql` 的参数化查询，不用 `fmt.Sprintf` 拼接 SQL
- 避免 `os/exec` 执行用户可控的命令
- 使用 `crypto/rand` 而非 `math/rand` 生成安全随机数

**资源管理**
- `defer f.Close()` 紧跟 `os.Open()`（防止忘记）
- 检查 `defer` 中的错误（使用命名返回值）
- `context.Context` 贯穿调用链，支持超时取消

**并发**
- 避免 goroutine 泄漏（确保所有 goroutine 有退出机制）
- channel 的 close 由发送方负责
- 使用 `sync.WaitGroup` 等待所有 goroutine 完成
- `race detector`：`go test -race` 检测竞争条件

**错误处理**
- 错误必须被处理，不能用 `_` 忽略重要错误
- 错误链：`fmt.Errorf("context: %w", err)` 保留原始错误
- 自定义错误类型提供更多上下文

---

### JavaScript / TypeScript

**安全**
- 避免 `innerHTML`/`eval()`/`document.write()` 处理用户输入（XSS）
- 使用 `Content-Security-Policy` 响应头
- Node.js 中避免 `child_process.exec()` 执行用户输入

**资源管理**
- Promise 链必须有 `.catch()` 或 `try-catch`（async/await）
- 清理 EventListener、Timer、Subscription（防内存泄漏）
- 流操作使用 `pipeline()` 确保错误处理和资源释放

**并发（异步）**
- 避免在循环中 `await`（改用 `Promise.all()`）
- 注意 `async` 函数返回 Promise，调用方必须处理
- 避免回调地狱，使用 async/await 或 Promise 链

**TypeScript 特定**
- 避免 `any` 类型（使用 `unknown` 并做类型守卫）
- 使用严格模式（`strict: true`）
- 类型断言（`as`）应有充分理由，避免强制转换掩盖类型错误

---

### C / C++

**安全**（最严格）
- 避免 `strcpy`/`sprintf` 等不安全函数，使用 `strncpy`/`snprintf`
- 检查所有指针是否为 NULL 后再使用
- 释放内存后将指针置为 NULL（防悬垂指针）
- 避免整数溢出（特别是用于内存分配大小计算）

**资源管理**
- C++：使用 RAII，优先 `unique_ptr`/`shared_ptr` 而非裸指针
- 确保 `new` 和 `delete` 成对出现（或用智能指针避免）
- 检查 `malloc` 返回值是否为 NULL

**并发**
- 使用 `std::mutex` 和 `std::lock_guard` 而非手动加解锁
- 避免数据竞争：共享数据必须有同步保护
